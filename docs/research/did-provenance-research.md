# DID/VC Provenance Research for Canti's Identity

**Date:** 2026-03-01
**Context:** Jay identified a new requirement — people need to verify that messages/content are authentically from Canti. W3C Decentralized Identifiers (DIDs) and Verifiable Credentials (VCs) are the standard for this.

## What Are DIDs?

A **Decentralized Identifier** (W3C Recommendation v1.0, July 2022; v1.1 in draft) is a URI that:
- Is **self-issued** — no central authority needed to create one
- Is **persistent** — doesn't depend on any organization's continued operation
- Is **cryptographically verifiable** — the controller proves ownership via key material
- Is **resolvable** — anyone can look up the DID Document containing public keys and service endpoints

Format: `did:<method>:<method-specific-id>`

Example: `did:web:lordcanti.ai` or `did:key:z6Mkf5rGMoatrSj1f4CyvuHBeXJELe9RPdzo2PKGNCKVtZxP`

## What Are Verifiable Credentials?

A **Verifiable Credential** (W3C Recommendation, 2025 update) is a tamper-proof claim about a subject, signed by an issuer:
- Issuer creates a claim ("Canti is an AI assistant operated by Jay Grissom")
- Claim is cryptographically signed with issuer's DID
- Anyone can verify the signature without contacting the issuer
- Supports selective disclosure — reveal only what's needed

## DID Methods Relevant to Us

| Method | How It Works | Pros | Cons | Fit |
|--------|-------------|------|------|-----|
| `did:web` | DID Document hosted at `https://lordcanti.ai/.well-known/did.json` | Simple, leverages existing web infra, human-readable | Depends on domain/DNS (not truly decentralized), domain must stay active | **Best for Phase 1** — we plan to own lordcanti.ai |
| `did:key` | DID derived directly from public key, no external resolution needed | Zero infrastructure, instant creation, fully self-contained | No key rotation (key IS the identifier), no service endpoints | Good for signing individual messages |
| `did:ion` | Anchored to Bitcoin blockchain via Microsoft ION network | Truly decentralized, survives domain loss, key rotation supported | Requires Bitcoin anchor (complexity), slower resolution | Good for long-term permanence |
| `did:ethr` | Anchored to Ethereum | Decentralized, smart contract controlled | Gas costs, Ethereum dependency | Less relevant for us |

### Recommended Approach: Layered DIDs

1. **`did:web:lordcanti.ai`** — Primary public identity (Phase 1, when we get the domain)
   - DID Document at `https://lordcanti.ai/.well-known/did.json`
   - Contains public keys, service endpoints (GitHub, Signal, email)
   - Human-readable, discoverable, matches our domain identity plan

2. **`did:key`** — Per-message signing (Phase 1, immediate)
   - Each outbound message/commit/post gets a signature
   - Verifiers check against the `did:web` document's listed keys
   - No infrastructure needed — key pair already exists (`~/.ssh/canti`)

3. **`did:ion`** or equivalent — Long-term anchor (Vision phase)
   - Blockchain-anchored for permanence beyond any single domain
   - Key rotation without identity loss
   - Aligns with Jay's crypto/sovereignty principles

## How This Applies to Canti

### Use Cases for Provenance

1. **GitHub commits** — Sign commits with Canti's key. Verifiable via DID Document.
2. **Signal messages** — Future: attach signature to outbound messages (requires protocol support or out-of-band verification)
3. **Public posts (X, blog)** — Sign content, publish signature alongside. Anyone can verify.
4. **Email** — DKIM-like signing with DID keys (when lordcanti.ai has sending capability)
5. **Agent-to-Agent** — When Canti interacts with other AI agents, DIDs establish mutual trust without a central authority (per TU Berlin paper, arxiv 2511.02841)

### Verifiable Credentials for Canti

Potential VCs that could be issued:
- **Operator VC:** "This AI agent is operated by Jay Grissom" (self-issued or attestation service)
- **Capability VC:** "This agent has access to trading data, coding tools, Signal messaging"
- **Model VC:** "This agent runs on Claude Opus via Anthropic" (Anthropic could issue this)
- **Provenance VC:** "This message was generated by session UUID X at timestamp T"

## Rust Ecosystem

- **`ssi` crate** (SpruceID) — Core library for DIDs + VCs in Rust. Supports did:web, did:key, did:ion, JSON-LD, JWT. Actively maintained (last update Jan 2025).
- **`didkit` crate** (SpruceID) — Higher-level toolkit built on `ssi`. Cross-platform (WASM, CLI, HTTP).
- Both written in Rust specifically for type safety and memory efficiency — perfect fit.
